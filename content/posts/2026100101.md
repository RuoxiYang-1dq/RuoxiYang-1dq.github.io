## 参数一：`min_tokens` —— 别让模型为你的废话买单

很多人在用 Qwen-Plus 做内容生成时，经常遇到一个情况：模型明明应该在 50 个字符内就能回答完的问题，非要拖到 200 多个字符。看似内容更丰富了，但你多付了 150 个Token的钱，得到了大量冗余信息。

这背后的问题出在哪？出在你没有设置**输出最短长度的限制**。

在[千聚api中转站](https://www.qianjuai.com/)的 API 文档中，`min_tokens` 这个参数，就是用来控制模型思考后输出结果的最短长度的。很多人只关注 `max_tokens`（最长输出），却忽略了 `min_tokens`。

举个例子，你问：“今年是哪一年？” 正常答案就 4 个字（含标点），Token 消耗极低。但如果你不设置 `min_tokens` 为 1，模型可能会默认输出“您好，今年是2024年，祝您生活愉快。”，凭空多出十几个Token。

**如何操作：**
在调用[千聚api中转站](https://www.qianjuai.com/) Qwen-Plus 时，直接设置 `“min_tokens”: 1`。这样模型在回答问题后，就不会画蛇添足地补充废话。如果任务需要一定的内容量，再适当调高这个数值。

**效果：**
对于“是/否”类、简短问答类任务，输出 Token 消耗量直接降低 50%-70%。

## 参数二：`stop` —— 精准截停，拒绝无效续写

这是很多人完全没接触过的一个参数，但它却是“降本大杀器”。

`stop` 参数，说白了就是告诉模型：“你输出到这儿，就打住，不许再往下写了。”

比如你在写代码或者填报表时，往往只需要模型输出一段完整的代码块或表格数据。但在不设置 `stop` 的情况下，模型写完代码后，可能还会继续输出“这段代码的思路是...”或者“请注意以上代码可能需要...”一堆毫无意义的后缀。

这些内容不仅耗费 Token，你还得再花时间去“清洗”输出结果。这等于花钱买垃圾数据。

**如何操作：**
在请求体中，加入 `“stop”: [“\n\n”, “.”, “!”]` 。你可以根据具体任务定制停止词。比如只想要一段代码，就把 `“stop”: [“\n\n”]`，模型一输出完代码块就立刻闭嘴。

**效果：**
假设你每次调用多支付 20 个 Token 的“后遗症”文本。每天调用 1000 次，一天就损失 20000 个 Token。设置 `stop` 后，这些 Token 的钱全部省下了，调用量越大，省得越多。

## 参数三：`response_format` —— 结构化输出，不做“清洗工”

很多人在调用 Qwen-Plus 做提取任务时，习惯让模型自由发挥，输出一段自然语言。然后，自己再写正则、写判断逻辑去提取关键信息。这种做法不仅消耗开发时间，还会因为模型输出格式不固定，导致每次调用消耗的 Token 数量波动巨大。

[千聚api中转站](https://www.qianjuai.com/)的 API 文档里明确提到，Qwen-Plus 支持 `response_format` 参数，你可以强制模型以 JSON 格式输出。

**举个例子：**
你想让模型提取“张三毕业于清华大学，现在在阿里巴巴工作”这句话里的姓名和公司。

- **不设置 response_format：**
  模型输出：“根据您提供的信息，该人物的姓名是张三，其当前就职的公司是阿里巴巴。”
  消耗 Token：23 个。

- **设置 response_format 为 JSON：**
  模型输出：`{“name”: “张三”, “company”: “阿里巴巴”}`
  消耗 Token：12 个。

**如何操作：**
在调用[千聚api中转站](https://www.qianjuai.com/)的 API 时，在 `messages` 数组里最后一条消息中加入 `“response_format”: {“type”: “json_object”}`，并在 System Prompt 里明确告诉模型：“你必须以JSON格式输出你的回答。”

**效果：**
输出长度固定、可预测，Token 消耗量直接下降 50%-60%。更重要的是，你不再需要编写后处理代码去“清洗”数据，开发效率翻倍。

json
{
  “model”: “gpt-4”,
  “messages”: [...],
  “stop”: [“\n”],
  “min_tokens”: 1,
  “response_format”: {“type”: “text”} // 代码生成不适合JSON，但我们可以通过stop和min_tokens配合限制输出

这样设置后，模型几乎只输出 5 行代码，Token 消耗量从 120 个直接降到 30 个。
