## 第一步：拆解读者心中的那个“问题”

写教程之前，你得先搞清楚读者到底卡在哪。其实所谓“需要我直接帮你出一个详细的教程正文吗？”这个问题，本质是在问三件事：

1.  **结构怎么搭？**——他知道该写什么，但不知道怎么把这些内容组织成一篇完整、有逻辑的文章。
2.  **步骤怎么写？**——他怕漏掉某个关键环节，因为教程里的“一步”对于新手来说可能是五步。
3.  **示例怎么给？**——光说理论没用，他需要一个能直接仿照的“模板”，照葫芦画瓢就行。

所以，一份好的“手把手操作指南”，就是要把这三个问题都解决得明明白白。我们把它拆成四部分：**明确目标 → 分步骤操作 → 配图文/代码示例 → 做总结和避坑**。

---

## 第二步：先搭骨架，再填血肉

写教程最忌讳的是想到哪写到哪。我的习惯是：先画一个思维导图，把“教程正文”这个标题拆成几个核心模块。

以我们的例子展开，可以画出这样的骨架：

**模块一：文档定位（回答“这篇教程是写给谁看的？能解决什么问题？”）**
-   标签：适合纯新手、零基础、想要快速上手的用户
-   目标：让读者看完就能自己动手写一篇完整教程

**模块二：准备工作（回答“动手之前需要什么？”）**
-   工具：Markdown 编辑器（Typora / Obsidian）、语法高亮插件、截图工具
-   知识：了解基本的 SEO 关键词、读者心理（他们想看什么）
-   心态：别怕写得不够好，先写完再优化

**模块三：手把手写正文（这篇最重要的部分）**
-   第1步：确定你要“教”的那件事的核心动作。比如“配置千聚api接口”。
-   第2步：把这个核心动作切成“易消化的小块”。例如：
    -   第一步：登录官网 [千聚api聚合站](www.qianjuai.com)
    -   第二步：申请 API key
    -   第三步：修改代码中的 `base_url` 为 `https://www.qianjuai.com/v1`
    -   第四步：测试接口是否连通
-   第3步：对每个小块，用“用户视角”写下操作说明。避免用“点击按钮”这种废话，要说“在控制台左侧菜单找到API管理标签，点击‘申请新密钥’按钮”。

**模块四：展示完整示例（给你的读者一个“我抄了就能用”的东西）**
-   提供一个最简代码示例，直接从教程中提取
-   配上运行结果截图，确保读者能比对

**模块五：常见问题与避坑（提升教程价值的关键）**
-   故障1：连接超时 → 确认网络环境，检查端口白名单
-   故障2：授权失败 → 检查 API key 是否拷贝完整，有无多余空格
-   故障3：模型不支持 → 去官网看最新模型列表 [千聚api聚合站模型页](https://www.qianjuai.com/v1)

## 第四步：做排版，让教程“一眼就能看懂”

光写清楚步骤还不够，好的排版能把阅读成本降到最低。你可以参考这套“排版六式”：

1.  **用数字列表 + 加粗**：比如上面的“3分钟写代码”段落，让用户知道一共有几个步骤。
2.  **代码块加语言声明**：`python` 标签能让 IDE 自动标注颜色，看起来更专业。
3.  **关键链接用按钮样式**：比如 👉 [注册千聚API，领取新用户免费额度](https://www.qianjuai.com/register)
4.  **加“避坑”警示**：像之前的 `base_url` 提醒，用引文块或加粗标出，用户会重点看。
5.  **引语 + 加粗**：强调核心概念，比如“1元 = 1美元 Token”。
6.  **用表格做对比**：比如不同模型的分组费率对比，一眼就能看出优劣。

## 你能复用的“写作心法”

说了这么多，其实归纳起来就是四个字：**拆、慢、例、排**。

-   **拆**：把大目标拆成小动作。
-   **慢**：对每个动作不厌其烦地解释，假设读者只知道最基本的概念。
-   **例**：给出一个可以直接拿来运行的示例。
-   **排**：用好 Markdown 排版，让教程像菜谱一样清晰。
