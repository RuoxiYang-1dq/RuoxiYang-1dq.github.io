## 第一步：搞清 401 的“真面目” —— 不是封号，是“对不上暗号”

401 Unauthorized 这个状态码，在 HTTP 协议里不是“封号”，它只是说：你发来的请求里，缺少或提供了错误的认证凭据。

常见的“假 401”场景包括：

1.  **你真的 Key 错了** —— 比如 Key 被截断、多了空格、复制了过期的旧 Key。
2.  **Key 对的，请求格式错了** —— 比如请求头里 `Authorization` 字段写成了小写，或者 Bearer 前缀写成了 `bearer`、`BEARER`。
3.  **环境变量没给对** —— 你的代码或客户端工具，读取了一个空的或者错误的变量。
4.  **Base URL 不匹配** —— 发请求的地址（API 端点）跟你 Key 对应的服务不一致。

在这些原因里，90% 的 401 都是第 2 点和第 4 点造成的。换句话说，你拿着一张对的卡，刷了错误的门禁系统。

## 第二步：3 分钟极速检修法 —— 改 1 个参数 + 配 2 处环境变量

下面这个流程，我已经在 3 个不同的项目里验证过，每次都能把 401 迅速拉回 200。总共就三步。

### 1. 改 1 个参数：Base URL 必须精准

这是最重要的一步。使用[千聚ai大模型中转站](https://www.qianjuai.com/)时，OpenAI 的 SDK 或者你的 HTTP 客户端里，`base_url` 必须是这个：


https://www.qianjuai.com/v1


不要写成 `https://api.openai.com/v1`，那肯定 401。也不要多加斜杠或漏掉 `v1`。很多朋友抄文档时，把 `base_url` 末尾多写了一个 `/`，比如 `https://www.qianjuai.com/v1/`，这种情况下，拼接出完整的请求路径会变成 `https://www.qianjuai.com/v1//chat/completions`，服务器不认识这个路径，直接返回 401 或 404。

**正确做法**：直接在代码里写：

python
base_url = "https://www.qianjuai.com/v1"


**如果你用 curl 或 Postman 工具**：在请求的 Base URL 处填入上述地址，路径部分写 `/chat/completions` 或其他所需端点。

### 2. 配 2 处环境变量：让你的 Key 被正确读取

你复制好了 API Key，但在服务器或工具里，它可能根本没被读取。

**第一处：系统的环境变量**

很多 AI 工具（比如 Cursor、LobeChat、ChatGPT Next Web）会自动读取环境变量 `OPENAI_API_KEY`。

-   在 Linux/Mac 终端里设置：
    bash
    export OPENAI_API_KEY="sk-你的千聚APIKey"
    
-   在 Windows 命令提示符里设置：
    cmd
    set OPENAI_API_KEY=sk-你的千聚APIKey
    
-   如果是在 Python 代码里加载 `.env` 文件，确保 `.env` 文件中有这一行：
    
    OPENAI_API_KEY=sk-你的千聚APIKey
    

**第二处：应用程序的配置界面**

对于自带 UI 的工具，比如沉浸式翻译、Cline 等，它们会在设置页有一个单独的 API Key 输入框。**这里填的 Key，优先级最高，会覆盖系统环境变量。**

这里有个很容易踩的坑：你在系统环境变量里设了一个旧 Key，又在工具设置里填了一个新 Key，但工具因为缓存或其他原因还在使用旧 Key。所以，填完新 Key 后，**务必重启工具或重启终端会话**。

### 3. 验证：发一个最简单的请求

做完上面两步，什么都不用想，直接发一个最小化请求来验证。

你可以用 curl 测试（Windows 建议用 Git Bash 或 WSL）：

bash
curl -X POST https://www.qianjuai.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-你的千聚APIKey" \
  -d '{
    "model": "gpt-3.5-turbo",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }'


如果返回 HTTP 200，并且 body 里包含 `choices` 字段，说明问题解决了。如果依然返回 401，检查 `Authorization` 头里的 Key 是否为最新，以及 `model` 参数是否支持。

## 第三步：针对常见“假 401”的特例排查

有时候，你按上面三步走完，依然 401。那可能是下面几种特殊情况：

-   **Key 格式不对**：千聚的 API Key 通常以 `sk-` 开头。确保没有复制到多余的空格或换行。在 `Authorization` 头里，格式必须是 `Bearer sk-具体字符串`。
-   **模型名字写错了**：如果你填的模型名（如 `gpt-3.5-turbo`）在该分组下不可用或已废弃，可能返回 401 或 404。可以在千聚的官网查看当前支持的模型列表。
-   **网络环境问题**：有些内网环境会劫持或修改 HTTP 请求头。尝试在纯公网环境（如手机热点）下测试。
-   **缓存冲突**：如果你用的是代理工具（如 `http_proxy` 环境变量），它可能缓存了错误的认证信息。临时取消代理再试一次。
