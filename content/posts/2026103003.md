## 为什么你的GPT-5.2调用成本总是降不下来

很多人觉得大模型调用就是“扔一段话进去，等它回复就行”。这个想法本身没错，但问题出在“扔”的方式上。

举几个最常见的烧钱姿势：
1. **对话历史越堆越长**——每次请求把之前所有对话都带上，一个会话跑100轮，光历史Token就上万。
2. **频繁重复调用**——同一个任务，比如数据清洗，非要每次单独请求，而不是批量处理。
3. **模型选择无脑最贵**——明明可以用低版本模型完成的任务，非要硬上满血版GPT-5.2。
4. **错误处理过于粗暴**——遇到超时或报错，直接重试整个请求，而不是优雅地兜底。

这些习惯累积下来，成本翻几倍是常事。接下来我直接上代码示例，看看最优写法长什么样。

## 核心优化一：砍掉冗余对话历史

这是最容易被忽视的烧钱点。很多人的代码长这样：

python
# 错误示例：每次请求都带上完整历史
messages = []
while True:
    user_input = input("你问：")
    messages.append({"role": "user", "content": user_input})
    response = client.chat.completions.create(
        model="gpt-5.2",
        messages=messages
    )
    assistant_msg = response.choices[0].message.content
    messages.append({"role": "assistant", "content": assistant_msg})
    print(f"GPT: {assistant_msg}")


跑50轮后，**messages**列表里就有100条消息，每次请求都要传输几千甚至上万Token。而实际上大多数对话的历史对当前问题没有任何帮助。

**优化方案**：只保留最近的几轮对话，或者只提取关键上下文作为提示。

python
# 优化示例：保留最近3轮对话
MAX_HISTORY = 6  # 3轮用户+3轮助手

messages = []
while True:
    user_input = input("你问：")
    messages.append({"role": "user", "content": user_input})
    if len(messages) > MAX_HISTORY:
        messages = messages[-MAX_HISTORY:]  # 只保留最近6条
    
    response = client.chat.completions.create(
        model="gpt-5.2",
        messages=messages
    )
    assistant_msg = response.choices[0].message.content
    messages.append({"role": "assistant", "content": assistant_msg})
    print(f"GPT: {assistant_msg}")


这样一改，每次请求的Token数从几千降到几百，成本直接减少70%以上，而且对话质量不会有明显下降——因为GPT-5.2本身就很擅长理解最近的上下文。

## 核心优化二：批量处理，减少API调用次数

**串行请求**是另一个吞金大户。假设你有1000条评论需要做情感分析，最笨的办法就是循环1000次调用：

python
# 错误示例：一条一条调

results = []
for review in reviews:
    response = client.chat.completions.create(
        model="gpt-5.2",
        messages=[{"role": "user", "content": f"分析情感：{review}"}]
    )
    results.append(response.choices[0].message.content)


1000次调用 = 1000次网络往返 + 1000次Token消耗。实际上GPT-5.2完全可以一次处理多条。

python
# 优化示例：批量输入
reviews = ["这是第一条评论", "这是第二条评论", ...]
batch_prompt = "请对以下每条评论进行情感分析（积极/消极/中性），每条结果用分号分隔：\n"
for i, review in enumerate(reviews):
    batch_prompt += f"{i+1}. {review}\n"

response = client.chat.completions.create(
    model="gpt-5.2",
    messages=[{"role": "user", "content": batch_prompt}]
)
results = response.choices[0].message.content.split("；")


一次调用搞定1000条评论，成本从1000次请求变成1次请求，降幅超过99%。当然，如果数据量特别大，可以分批次，每批50-100条，但依然比串行划算得多。

## 核心优化四：优雅的错误处理与重试

很多人的错误处理是“挂了就重新调一次”，这等于把报错过的那次请求白花了。更优的做法是：用更短的超时时间、用指数退避重试，并且在重试时减少返回Token数。

python
import time
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
def call_gpt_with_retry(prompt: str, max_tokens: int = 2000) -> str:
    try:
        response = client.chat.completions.create(
            model="gpt-5.2",
            messages=[{"role": "user", "content": prompt}],
            max_tokens=max_tokens,
            timeout=15  # 15秒超时，避免长等待
        )
        return response.choices[0].message.content
    except Exception as e:
        print(f"请求失败，重试中... 错误：{e}")
        raise  # 触发重试


这样即使偶尔超时，也只浪费15秒和少量Token，而不是白白烧掉完整请求的配额。

## 综合案例：一个降本80%的完整Python脚本

下面是一个真实场景的示例：批量分析电商评论，并生成摘要报告。

python
import hashlib
from openai import OpenAI
from tenacity import retry, stop_after_attempt, wait_exponential

client = OpenAI(
    api_key="从千聚获取的key",
    base_url="https://www.qianjuai.com/v1"
)

CACHE = {}
BATCH_SIZE = 20

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
def batch_analyze(reviews: list[str]) -> list[str]:
    cache_key = hashlib.sha256("".join(reviews).encode()).hexdigest()
    if cache_key in CACHE:
        return CACHE[cache_key]
    
    prompt = "请对以下每条评论进行情感分析（积极/消极/中性），每条结果用【分号】分隔：\n"
    for i, review in enumerate(reviews):
        prompt += f"{i+1}. {review}\n"
    
    response = client.chat.completions.create(
        model="gpt-4o",  # 用中等模型，便宜够用
        messages=[{"role": "user", "content": prompt}],
        max_tokens=1000,
        timeout=30
    )
    result = response.choices[0].message.content
    results = result.split("【分号】")
    CACHE[cache_key] = results
    return results

def main():
    all_reviews = ["评论1", "评论2", ...]  # 假设有1000条
    for i in range(0, len(all_reviews), BATCH_SIZE):
        batch = all_reviews[i:i+BATCH_SIZE]
        batch_results = batch_analyze(batch)
        print(f"批次 {i//BATCH_SIZE + 1} 处理完成")
    
    # 生成摘要时用满血版GPT-5.2，只调一次
    summary_prompt = f"请根据以下分析结果生成一份摘要报告：{str(batch_results)}"
    summary = client.chat.completions.create(
        model="gpt-5.2",
        messages=[{"role": "user", "content": summary_prompt}],
        max_tokens=500,
        timeout=30
    )
    print(summary.choices[0].message.content)

if __name__ == "__main__":
    main()


这个脚本相比原始写法：批量调用代替串行、缓存避免重复请求、中等模型处理常规任务仅在最后用满血版、加超时和重试——综合下来成本骤降80%以上，你亲自跑一遍就能体会。
