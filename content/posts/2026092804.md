## 错误二：忽视错误处理，接口返回502还以为是代码写错了

跑通之后，我开始得意了。直接写了个循环，批量处理对话数据。结果跑到第10条，程序崩了。报错信息是 `502 Bad Gateway`，我第一反应：“我代码里参数传错了？”

实际上，Grok3mini的API在高并发下偶尔会不稳定。如果你不做重试和错误捕获，一条502就能让整个任务报废。

正确的做法是引入重试机制，并且捕获 `openai.RateLimitError` 和 `openai.APIConnectionError`：

python
import time
from openai import OpenAI, RateLimitError, APIConnectionError

client = OpenAI(base_url="https://www.qianjuai.com/v1", api_key="你的Key")
model = "grok-3-mini"

def chat_with_retry(messages, max_retries=3):
    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model=model,
                messages=messages
            )
            return response.choices[0].message.content
        except (RateLimitError, APIConnectionError) as e:
            print(f"第{attempt+1}次重试，错误信息：{e}")
            time.sleep(2 ** attempt)  # 指数退避
    raise Exception("重试失败，程序终止")


加了这个重试逻辑之后，即使偶尔遇到502，程序也能自动恢复。**没有这个，代码就像没系安全带的飙车。**

## 错误三：不用流式输出，傻等完整响应，多花3倍冤枉钱

这个是我的血泪之痛。我一开始为了代码简单，用的是非流式请求，就是等模型把所有Token生成完了，一次性拿回完整结果。

python
# 错误示例：非流式请求
response = client.chat.completions.create(
    model="grok-3-mini",
    messages=[{"role": "user", "content": "给我写一篇3000字的文章"}],
    stream=False  # 这是默认值
)


你们猜发生了什么？这个请求挂了足足1分20秒才返回，消耗了约800个Token。但是！**API是按输出Token计费的，而流式请求和非流式请求的输出Token是一样的，但等待时间和网络稳定性完全不一样。**

更致命的是，非流式请求的超时设定通常比较短（比如30秒）。Grok3mini生成长文本时，如果30秒内没完整返回，客户端就判定超时，但模型其实已经生成了500个Token，这些Token的费用照样扣。**你没拿到结果，钱却已经花了。**

正确做法是用流式：

python
# 正确示例：流式请求
response = client.chat.completions.create(
    model="grok-3-mini",
    messages=[{"role": "user", "content": "给我写一篇3000字的文章"}],
    stream=True  # 关键：流式
)

collected_content = ""
for chunk in response:
    if chunk.choices[0].delta.content:
        content_part = chunk.choices[0].delta.content
        collected_content += content_part
        print(content_part, end="")

print(f"\n总输出字数为: {len(collected_content)}")


流式请求的好处是：Token一边生成一边接收，即使中途断网，你也不会损失整个生成的费用；而且你可以实时看到内容产出，用户体验好太多了。**不流式，你就是在烧钱买焦虑。**
