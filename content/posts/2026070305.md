## 避雷第二式：`model` 名字没写对，你在用“天价通道”

很多人觉得，Node.js 里写 `model: 'gpt-4o'` 不就完了。

但你有没有想过，**你写的模型名，决定了后台走的是哪条渠道**

现在很多中转站，同一个 `gpt-4o` 名字背后是多个渠道池的。你如果没搞清楚对应关系，平台会默认给你走一条“稳但贵”的通道。比如你在[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)，默认混合分组是官方价格的 1 倍，但如果你没注意渠道选择，代码里却写了个特定的逆向模型名称，它可能走了官转通道——费率直接跳到 3 倍甚至更高。

举个常见的“冤枉钱”写法：

javascript

  model: 'claude-3-opus-20240229', // 这是官方名

## 避雷第三式：别忽视 `stream` 选项的“隐藏成本”

很多人 Node.js 调用 ChatGPT 写对话，喜欢直接加 `stream: true`，以为这样能节省算力。确实，流式输出用户体验好，而且能减少超时问题。

但你可能不知道，**部分 API 中转站的定价策略里，流式输出的计费逻辑跟非流式不同**。

怎么不同法？我见过这么一种情况：

你写 `stream: true`，平台以为你要做实时对话，会给你分配最贵的高并发通道，按每字节计费；而如果你写 `stream: false`，它走的是普通批量推理通道，按 token 计费，单价更低。

虽然[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)没有这种区别定价，但如果你在写 Node.js 代码时，不加思考地复制别人的 `createChatCompletion` 示例，很可能就带上了 `stream: true`。而你的场景其实不需要流式（比如批量处理历史对话），那你就白付了这部分“通道费用”。

**怎么避？**

明确你的需求。如果是后台异步处理，直接写 `stream: false` 甚至直接不传这个参数，让它走默认的非流式通道。示例代码不需要一味求新求全。

javascript
// 正确写法：不传 stream，走默认低价通道
const response = await client.chat.completions.create({
  model: 'gpt-4o-mini',
  messages: [{ role: 'user', content: '解释一下什么是Token' }],
  max_tokens: 300,
});


这样写，不仅代码简洁，计费也对你最有利。
